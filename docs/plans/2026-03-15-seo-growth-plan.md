### What's Working

1. **Queue time pages are SEO gold.** Mine Cart Madness and Ministry are the #2 and #4 organic landing pages with strong engagement. These pages rank because they're deeply data-driven, well-written, and target low-competition keywords.

2. **Crowd calendars are the #1 organic entry point.** The Universal crowd calendar gets 316 organic sessions — more than any other page. This matches competitor data: TouringPlans' #1 SEO pages are crowd calendars.

3. **Epic Universe content dominates.** 8 of the top 10 pages are Epic Universe content. The park is new (opened 2025) and no competitor has fully established topical authority yet.

4. **Growth trajectory is exponential.** The site is riding a wave of Epic Universe demand that hasn't peaked yet.

5. **ChatGPT referrals are real.** 198 users from ChatGPT means the content is being cited by AI — this will only grow.

### What's Not Working

1. **Spring break content has 62% bounce rate.** Despite high traffic (1,184 views), the spring break guide isn't retaining users. Likely too long/dense for mobile users (81% are on mobile).

2. **Strategy guide has 53% bounce rate.** The #1 page by views loses over half its visitors immediately.

3. **Organic search is under-indexed.** Only 22% of traffic comes from search. Competitors like TouringPlans get 50-70% from search. The site is over-reliant on social virality.

4. **57% of pages missing meta descriptions.** 25 of 44 public pages have no meta description — this directly impacts SERP click-through rates.

5. **Fragmented information architecture.** Two parallel hierarchies exist:
   - `/parks/epic-universe/` — park page with strategy
   - `/universal-orlando/epic-universe/` — queue times hub
   - `/orlando/spring-break-2026/` — Orlando-level content

   This splits link equity and confuses Google about which page to rank.

6. **No individual ride pages optimized for search.** Queue time pages exist but only for 6 rides. TouringPlans has a page for EVERY ride. Each ride page is a separate search opportunity.

7. **Limited schema markup.** No Organization, SoftwareApplication, or TouristAttraction schema. Missing structured data means missing rich results in SERPs.

8. **Generic og:image on 80% of pages.** Social shares show the same logo.png instead of page-specific images — reduces CTR on social and in Google Discover.

- Navigation menus (header/footer)
- Any cross-links in content body text

#### 1.1.1 Canonical URL Rules

All canonical URLs on the site must follow these rules to prevent duplicate URL surfaces:
- **Protocol:** Always `https://` (never `http://`)
- **Host:** Always `rideready.app` (never `www.rideready.app`)
- **Trailing slash:** Always include trailing slash for directory-style URLs (e.g., `/parks/epic-universe/`, not `/parks/epic-universe`)
- **Canonical tags:** Every page's `<link rel="canonical">` must use the full absolute URL following these rules

#### 1.2 Execute URL Migration `[EU]`

**GitHub Pages constraint:** GitHub Pages does not support server-side 301 redirects. All redirects must use client-side HTML redirect pages (meta refresh + canonical tag + JS fallback). These return HTTP 200 but Google treats `<meta http-equiv="refresh" content="0; url=...">` combined with `<link rel="canonical">` pointing to the destination as equivalent to a 301 for ranking purposes.

**Redirect page template** (place at old URL path as `index.html`):
```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta http-equiv="refresh" content="0; url=[DESTINATION]">
  <link rel="canonical" href="https://rideready.app[DESTINATION]">
  <title>Redirecting...</title>
</head>
<body>
  <p>Redirecting to <a href="[DESTINATION]">[DESTINATION]</a>...</p>
  <script>window.location.replace('[DESTINATION]');</script>
</body>
</html>
```

For each redirect in the ownership table:
1. Create the destination page (merge content if needed)
2. Replace old URL's `index.html` with the redirect template above
3. Update ALL internal links sitewide that point to the old URL
4. Update sitemap.xml (remove old URL, ensure destination is listed)
5. Update navigation menus

**Verification gate:**
- `grep -r` across all non-redirect HTML files for links to old URL paths: zero matches = pass
- Each redirect page contains both `<meta http-equiv="refresh">` and `<link rel="canonical">` pointing to the correct destination
- Sitemap.xml contains zero redirected URLs

#### 1.3 Implement Breadcrumb Navigation on All Pages `[EU]`

Add visible breadcrumbs + BreadcrumbList schema. Example:
`Home > Parks > Epic Universe > Rides > Mine-Cart Madness`

#### 1.4 Add "Related Content" Cross-Links `[EU]`

Every EU page should link to 3-5 related EU pages. Queue time pages link to other queue time pages and the rides hub. Crowd calendar links to strategy guide. Park hub links to all sub-pages.

### Phase 3: Content Expansion — Attraction Pages (3-5 days)

This is the highest-ROI content investment. Each page is a new organic search entry point.

#### 3.0 Attraction Page Types and Inclusion Criteria

Not all Epic Universe attractions are traditional rides with queues and wait times. The plan defines three page types based on attraction characteristics:

**Type A: Ride Pages (wait-time-driven)**
- **Criteria:** Has a queue, collects wait time data, has height requirements
- **Template:** Full data-driven page with wait time charts, best time to ride, queue experience
- **Examples:** Mine-Cart Madness, Mario Kart, Stardust Racers, Ministry of Magic, Monsters Unchained, Hiccup's Wing Gliders
- **Sections:** Quick Stats, Average Wait Times by Hour, Best Time to Ride, Queue Experience, Touring Tips, Is It Worth the Wait?, What to Ride Next, FAQ

**Type B: Attraction Pages (experience-driven)**
- **Criteria:** Has a queue but limited/no meaningful wait time data (walk-throughs, slow-loaders, interactive experiences). OR has wait time data but too sparse to publish responsibly (see 3.0.1 below).
- **Template:** Lighter page focused on experience description, tips, and accessibility. No wait time charts. Replaces "Average Wait Times" section with "What to Expect" narrative section.
- **Examples:** Yoshi's Adventure (omnimover — consistent low waits), walk-through attractions
- **Sections:** Quick Stats (duration, height req, thrill level), What to Expect, Touring Tips, Accessibility, What to Do Next, FAQ

**Type C: Show/Entertainment Pages (schedule-driven)**
- **Criteria:** No queue, scheduled performances or ambient entertainment
- **Template:** Schedule-focused page with show times, seating tips, and content description. No wait time data. No height requirement.
- **Examples:** The Frog Choir, any stage shows, street entertainment
- **Sections:** What It Is, Show Schedule, Best Seats / Viewing Tips, What to Do Next, FAQ

**All three types share:** TouristAttraction schema, FAQPage schema, BreadcrumbList schema, cross-links, and consistent visual design.

#### 3.0.1 Minimum Data-Coverage Standard for Publishing Wait Time Claims

Before any page publishes "average wait times," "best time to ride," or data-driven recommendations, the underlying data must meet these thresholds:

| Data Claim | Minimum Requirement |
|-----------|-------------------|
| "Average wait time" (any time period) | At least 500 data samples for that time period |
| "Best time to ride" | At least 14 full operating days of hourly data |
| "Wait times by hour" chart | At least 7 full operating days of data per hour slot |
| "Peak wait time" / "Record wait" | At least 30 operating days of data |

**If an attraction doesn't meet these thresholds:**
- Publish as Type B (experience-driven) regardless of ride type
- Add a notice: "We're still collecting data for this attraction. Check back soon for wait time analysis."
- Promote to Type A once thresholds are met

**Verification:** Before Phase 3 begins, audit each attraction's data coverage against these thresholds. This determines which attractions get Type A vs Type B pages at launch.

#### 3.1 Create Individual Attraction Pages for ALL Epic Universe Attractions `[EU]`

Currently only 6 queue time pages exist. Epic Universe has 19+ rides/attractions. Create pages for the remaining ~13, using the appropriate page type from 3.0.

**Step 1:** Audit all EU attractions and classify each as Type A, B, or C.
**Step 2:** Audit data coverage for each Type A candidate against thresholds in 3.0.1. Demote to Type B if thresholds not met.
**Step 3:** Create pages in priority order — Type A first (highest SEO value), then Type B, then Type C.

**URL pattern for all types:** `/parks/epic-universe/rides/[slug]/`
(Using "rides" in the URL even for shows/attractions is intentional — it matches user search behavior. Users search "[attraction] ride" even for non-rides.)

**Title pattern:**
- Type A: `[Name] Wait Times, Tips & Strategy | Ride Ready`
- Type B: `[Name] — Guide, Tips & What to Expect | Ride Ready`
- Type C: `[Name] — Show Times, Tips & Guide | Ride Ready`

#### 3.2 Create an All-Rides Hub Page `[EU]`

URL: `/parks/epic-universe/rides/`
Lists all attractions organized by world/land, linking to each individual page.

**Wait time display on hub page:** Since this is a static site, the hub page cannot show live/real-time wait times. Instead:
- Display **static "typical wait" ranges** based on historical data (e.g., "Usually 30-60 min") for Type A attractions
- Display "Walk-On" or "Minimal Wait" for Type B attractions
- Display "See Schedule" for Type C attractions
- Include a prominent CTA: "Get live wait times in the Ride Ready app"
- Add `article:modified_time` and update whenever data is refreshed

This avoids the stale-data problem of claiming "current" wait times on a static page.

### Phase 4: Content Expansion — Editorial Layer (Ongoing)

#### 4.0 Static Site Scalability Note

This plan roughly doubles the page count (from ~44 to ~80+). The current workflow is fully manual (hand-editing HTML files with inline CSS). At this scale, consistency risks rise: meta descriptions, schema markup, cross-links, and breadcrumbs must be maintained by hand across every page.

**Mitigation (not blocking — can be addressed incrementally):**
- Create a checklist/template for new page creation (required meta tags, schema blocks, cross-links, sitemap entry)
- Consider a lightweight static site generator (e.g., 11ty, Hugo) if manual maintenance becomes a bottleneck past 100 pages. This is a future decision, not a prerequisite.
- For now, use `grep -r` audits after each batch of new pages to verify consistency (e.g., all pages have meta description, canonical, schema).

#### 4.1 News/Updates Section (Move to Early — Parallel with Phase 3) `[EU]`

**Moved up from Week 7-8 to Week 3-4.** Freshness is a core part of the search strategy. Operational news captures time-sensitive queries ("[ride] closed", "[park] news") that no amount of evergreen content addresses. This capability should launch as early as possible.

Create a lightweight updates section at `/updates/` for fast-turnaround operational content:
- Ride closures and reopenings
- New attraction announcements
- Special event schedules
- Wait time trend reports ("March 2026 Wait Times: What We Learned")

**Structure:**
- `/updates/index.html` — Index/archive page listing all updates reverse-chronologically with dates, titles, and summaries. This is the crawl entry point.
- `/updates/[slug]/index.html` — Individual update pages with full content
- Each update page includes: `article:published_time`, `article:modified_time`, `datePublished` in Article schema, BreadcrumbList schema
- Add each new update URL to sitemap.xml with `<lastmod>` and `<changefreq>daily</changefreq>`
- Link to relevant updates from affected attraction/hub pages (e.g., a Stardust Racers closure update gets linked from the Stardust Racers ride page)
- The EU hub page (`/parks/epic-universe/`) should display the 3 most recent updates as a "Latest News" section

**Lifecycle for year-bound content:** Pages like spring break and holiday guides should follow a rollover policy:
- When the next year's version is created, add `rel="canonical"` on the old year pointing to the new year
- Add a banner to the old page: "Looking for [Year+1]? See our latest guide."
- Remove old year from sitemap.xml but keep the page live (preserves inbound links)

## Part 6: Design & Frontend Considerations

All new pages and improvements should follow the existing Ride Ready design system:
- Dark theme with coral/orange gradient accents (#FF416C → #FF4B2B, 45deg)
- Mobile-first (81% of users)
- 44px minimum tap targets
- Reduced motion support
- Focus-visible states
- Inline CSS (no external stylesheets)
- Google Analytics (G-KGYGTPMTQ3) on all pages

### Design Improvements Needed

1. **Sticky mobile TOC for long-form content** — Strategy guide and spring break guide need navigation help on mobile
2. **Collapsible data sections** — Dense tables should collapse on mobile with "tap to expand"
3. **Visual data cards** — Replace some text-heavy sections with scannable card layouts (wait time, crowd level, recommendation)
4. **Progress indicator** — For long guides, show reading progress
5. **Page-specific hero images** — Replace generic gradients with park/ride imagery where possible
